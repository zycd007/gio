## 一、需求背景

### 1.1 业务背景

GIO 设计事务所作为专业设计机构，需要通过社交媒体持续输出优质内容来：
- 展示设计案例，建立品牌专业形象
- 吸引潜在客户，扩大业务来源
- 增强与用户互动，提升品牌影响力

小红书作为国内领先的生活方式分享平台，是设计类内容的重要传播渠道。但高质量的小红书文案创作存在以下痛点：

### 1.2 用户痛点

| 痛点 | 描述 |
|------|------|
| **创作耗时** | 每条文案需要构思标题、正文、标签，耗时 30-60 分钟 |
| **风格把握难** | 小红书平台特有的"种草风"需要专门学习才能掌握 |
| **内容同质化** | 难以持续产出有差异化、有吸引力的内容 |
| **运营效率低** | 批量运营多个项目时，文案创作成为瓶颈 |

### 1.3 产品价值

通过 AI 大模型能力，实现：
- **效率提升**：将单条文案创作时间从 30+ 分钟缩短至 1 分钟内
- **质量保障**：基于最佳实践模板，确保文案符合小红书平台调性
- **风格多样**：提供多种文案风格选择，满足不同场景需求
- **批量生产**：支持快速生成多个项目的文案，提升运营效率

### 3.2 功能详细设计

#### 3.2.1 功能入口

**位置**：管理后台 -> 项目管理 -> 项目编辑弹窗

**入口设计**：
- 在项目编辑弹窗的底部，保存按钮旁边添加"AI 生成文案"按钮
- 按钮样式：渐变紫色背景（区别于主操作按钮），带 AI 图标
- 按钮文案："✨ AI 生成文案"

**触发条件**：
- 项目必须已保存（有项目 ID）
- 项目必须有至少一张图片（用于 AI 理解项目内容）

#### 3.2.2 风格选项

| 风格 | 名称 | 描述 | 示例语气 |
|------|------|------|----------|
| style_professional | 专业风 | 突出设计专业性和技术亮点 | "斩获国际大奖的极简主义力作..." |
| style_seed | 种草风 | 强调体验和感受，引发向往 | "真的被惊艳到了！这个设计太绝了..." |
| style_story | 故事风 | 讲述设计背后的故事 | "三年前，业主找到我们说..." |
| style_minimal | 极简风 | 简洁有力，突出核心信息 | "位置｜面积｜设计｜实景" |

#### 3.2.3 生成内容结构

生成的文案包含以下部分：

```
【标题】（15-25 字，吸引眼球）

【正文】（200-400 字，分段展示）
- 开篇：引入项目背景/亮点
- 中段：设计特色/空间亮点
- 结尾：引导互动/价值总结

【标签】（5-10 个相关话题标签）
#室内设计 #装修灵感 #极简主义 #设计案例 ...
```

#### 3.2.4 操作流程

```
1. 用户在项目列表中点击"编辑"按钮
2. 在项目编辑弹窗中，点击"✨ AI 生成文案"按钮
3. 弹出文案生成对话框
4. 选择文案风格（默认"种草风"）
5. 点击"开始生成"按钮
6. 显示加载状态（预计 5-10 秒）
7. 展示生成结果（标题、正文、标签）
8. 用户可编辑内容或点击"复制文案"
9. 点击"完成"关闭对话框
```

### 4.4 后端接口设计

#### 4.4.1 接口定义

**路径**：`POST /api/admin/projects/{projectId}/ai-copywriting`

**请求参数**：
```json
{
  "style": "style_seed"  // 文案风格
}
```

**响应格式**：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "title": "标题内容",
    "content": "正文内容",
    "tags": ["标签 1", "标签 2"],
    "generatedAt": "2026-04-02T10:30:00"
  }
}
```

**错误处理**：
```json
{
  "code": 500,
  "message": "AI 服务暂时不可用，请稍后重试",
  "data": null
}
```

在 `gio-admin` 模块中新增：

```
gio-admin/
```

## 六、数据设计

### 6.1 数据存储需求

| 数据项 | 是否存储 | 说明 |
|--------|----------|------|
| 生成的文案 | 否（V1.0） | 首期不存储，用户复制即走 |
| 生成历史 | 否（V1.0） | 首期不做，后续迭代考虑 |
| 调用日志 | 是 | 记录 API 调用用于成本核算 |

### 6.2 调用日志表设计

**表名**：`ai_copywriting_log`

```sql
CREATE TABLE `ai_copywriting_log` (
  `id` INT PRIMARY KEY AUTO_INCREMENT,
  `project_id` INT NOT NULL COMMENT '项目 ID',
  `style` VARCHAR(50) NOT NULL COMMENT '文案风格',
  `status` TINYINT NOT NULL COMMENT '状态：0-失败 1-成功',
  `tokens_used` INT DEFAULT NULL COMMENT '消耗 Token 数',
  `error_message` VARCHAR(500) DEFAULT NULL COMMENT '错误信息',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_project (`project_id`),
  INDEX idx_created (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='AI 文案生成调用日志';
```

### 6.3 日志记录时机

| 时机 | 记录内容 |
|------|----------|
| 调用 AI API 前 | project_id, style, status=PROCESSING |
| 收到响应后 | 更新 status, tokens_used, error_message |
